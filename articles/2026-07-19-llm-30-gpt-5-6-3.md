## 背景：なぜLLMが数学の未解決問題に挑めるのか

2026年に入り、大規模言語モデルが「純粋数学」の領域で成果を出すニュースが相次いでいる。DeepMindのAlphaProof、OpenAIのo3系モデルによる数学オリンピック問題への取り組みは記憶に新しいが、今回注目されているのは**GPT-5.6が凸最適化（Convex Optimization）の領域で30年来のギャップを埋める証明アプローチを示した**という報告だ。

凸最適化は機械学習・オペレーションズリサーチ・制御工学の基盤となる数学分野で、「局所最適＝大域最適」が保証される凸関数に対して効率的に最適解を求める理論体系を扱う。1990年代のBoyd & Vandenberghe以来、内点法や劣勾配法の理論は成熟しているが、特定のサブクラス（準凸関数族のある収束レート証明など）では未解決の命題が残っていた。

LLMがこの種の問題に取り組めるようになった背景には、以下の3つの技術的変化がある。

### 1. スケールと推論能力の非線形的な跳躍

GPT-4からGPT-4oを経てo1/o3/GPT-5系列へと続く進化の中で、「パラメータ数の増加」よりも**テスト時計算量（Test-Time Compute）の増加**が決定的な役割を果たした。o1以降のモデルは推論時に内部的なチェーン・オブ・ソート（Chain-of-Thought）を自律的に長く伸ばす能力を持ち、これが数学的な多段論証に直結する。

```
一般的なLLM推論:
入力 → 1回のフォワードパス → 出力

o1/o3系のLLM推論:
入力 → [think: ステップ1の試行 → 自己評価 → 修正 → ステップ2の試行 → ...] → 出力
```

この「内部ループ」は、ヒューマンの数学者が「試してみて、ダメなら別の補題を探す」という探索プロセスに構造的に似ている。

### 2. ベリファイア（Verifier）との組み合わせ

LLM単体では「それっぽい証明」と「正しい証明」を区別できないという問題がある。最近の研究では、**LLMをProposer（命題・証明ステップの生成器）として、形式検証ツール（Lean 4・Coq等）をVerifier（正しさの検証器）として組み合わせる**ハイブリッドアーキテクチャが主流になりつつある。

```
[LLM Proposer]
  ↓ 証明ステップ候補を複数生成（ビームサーチ的）
[Lean 4 / Coq Verifier]
  ↓ 形式的に正しいステップのみを通過させる
[LLM Proposer]
  ↓ フィードバックを受けて次ステップを生成
  ...（繰り返し）
[完成した形式証明]
```

今回のGPT-5.6のケースが純粋にLLMだけで達成したのか、このようなVerifierとのループを用いたのかは詳細を要確認だが、少なくとも「プロンプト設計でここまでできる」という事実は重要だ。

### 3. 数学的コーパスの質的変化

2022年以降、arXiv・Lean MathLib・IMO Grand Challenge等の**形式証明ライブラリ**がLLMの学習データとして大量に含まれるようになった。自然言語の「証明っぽい文章」ではなく、機械検証可能な形式証明を大量に学習したことで、モデルが「論理的に正しい推論ステップ」のパターンを内部表現として獲得しつつある。

## 凸最適化とは何か：エンジニア向け最短説明

記事の後半でプロンプト設計の話をするために、凸最適化の基礎を簡単に整理しておく。

### 凸関数・凸集合の定義

関数 $f: \mathbb{R}^n \to \mathbb{R}$ が**凸（convex）**であるとは、定義域内の任意の2点 $x, y$ と $t \in [0, 1]$ に対して以下が成立することをいう：

$$f(tx + (1-t)y) \leq tf(x) + (1-t)f(y)$$

直感的には「弦が関数グラフより上にある」＝「お椀型」の関数だ。

```
凸関数 f(x) の例:
  - f(x) = x²
  - f(x) = e^x
  - f(x) = -log(x)  (x > 0)
  
非凸関数の例:
  - f(x) = sin(x)
  - f(x) = x³
  - ニューラルネットワークの損失関数（一般に非凸）
```

### なぜ「凸」が重要なのか

凸最適化問題の最大の利点は、**局所最適解が即ち大域最適解であること**が保証される点だ。つまり、勾配降下法のようなシンプルなアルゴリズムが確実にグローバル最適解へ収束する。

機械学習の損失関数はほぼ非凸だが、サポートベクターマシン（SVM）のヒンジ損失最小化・L1/L2正則化問題・ポートフォリオ最適化・最短経路問題などは凸最適化として定式化できる。

### 30年来のギャップとは

今回のGPT-5.6が取り組んだとされる問題は、**準凸関数（Quasiconvex Function）族に対する収束レートの証明**に関わるものだ。準凸関数とは「すべての下位集合 $\{x: f(x) \leq c\}$ が凸集合」という弱い条件しか満たさない関数クラスで、凸関数の一般化にあたる。

1990年代初頭に提唱されたある特定のアルゴリズム（準凸最小化の反復劣勾配法の変種）について、理論的な収束レートの上界が「経験的に速い」ことは知られていたが、厳密な証明が欠けていた、という文脈が報告されている。

LLMがここで果たした役割は「自力で全証明を発見した」というより、**複数の既存補題を組み合わせる新たな接続を見つけた**ことに近いと考えられる。これはまさにLLMの得意領域だ。

## プロンプト設計で数学的タスクの精度を上げる3パターン

「GPT-5.6がすごい」で終わらせず、エンジニアが今日から実務で使えるレベルに落とす。

### パターン1：`/goal` による目標明示型プロンプト

Fable 5 vs GPT-5.6 Solの比較検証でも話題になっていた `/goal` 記法は、LLMに**「何を達成すれば終わりか」を明示的に伝える**手法だ。

```
# /goal なし（漠然とした依頼）
「この行列が正定値かどうか確認して」

# /goal あり（目標明示型）
「/goal: 以下の3×3行列が正定値（Positive Definite）であることを証明せよ。
証明には固有値の計算またはシルベスターの基準を使うこと。
各ステップを明示し、最後に Yes/No で結論を出すこと。

行列:
[4, 2, 1]
[2, 3, 0]
[1, 0, 2]
」
```

**なぜ効くのか：** LLMは「回答」を生成するより「目標への到達」を最適化する方がステップの整合性が高くなる傾向がある。これはRLHF（人間フィードバック強化学習）の設計上、「完了基準が明確なタスク」のほうがリワードを計算しやすいため、学習時に優先されやすいからと考えられる。

### パターン2：補題分割（Lemma Decomposition）

大きな命題を一度に証明させようとすると、LLMは途中でステップを飛ばしたり、暗黙の仮定を置いたりして誤った証明を生成しやすい。**補題（Lemma）に分割して逐次確認する**パターンが有効だ。

```python
# 補題分割の擬似コード（Python + LLM API）

lemmas = [
    "補題1: f(x) が凸関数ならば、すべての下位集合 S_c = {x | f(x) ≤ c} は凸集合である",
    "補題2: S_c が凸集合ならば、その内部のどの2点を結ぶ線分も S_c に含まれる",
    "補題3: 上記2補題を用いて、f の最小化問題が局所最適=大域最適を満たすことを示せ",
]

proof_steps = []
for lemma in lemmas:
    prompt = f"""
    以下の補題を証明してください。証明は厳密で、各ステップに根拠を付けること。
    
    補題: {lemma}
    
    前提として確立済みの結果:
    {chr(10).join(proof_steps) if proof_steps else "なし"}
    """
    result = llm.complete(prompt)
    proof_steps.append(f"[確立済み] {lemma}: {result}")
```

**効果：** 各補題が独立したコンテキストで検証されるため、後段の補題が誤った前提を引き継ぐリスクが下がる。また、形式検証ツールとのインテグレーションポイントも補題単位で設けやすい。

### パターン3：反証探索との組み合わせ（Devil's Advocate）

LLMは「証明を完成させようとする」バイアスがある。つまり、実際には成立しない命題に対しても「それっぽい証明」を生成してしまうことがある。これを防ぐには**別のLLM呼び出し（または同一LLMへの別プロンプト）で反証を探させる**手法が有効だ。

```
# Proverプロンプト
「以下の命題を証明せよ: [命題P]」

# Devil's Advocateプロンプト（並列実行）
「以下の命題が**誤り**だと仮定した場合、反例を構築せよ。
もし反例が構築できないなら、その理由を説明せよ: [命題P]」

# 判定ロジック
- Proverが証明を提示 AND Devil's Advocateが反例を提示 → 矛盾・要人間確認
- Proverが証明を提示 AND Devil's Advocateが反例を見つけられない → 証明候補として採用
- Proverが証明を提示できない AND Devil's Advocateが反例を提示 → 命題が偽の可能性
```

このパターンは数学に限らず、**コードレビュー・セキュリティ監査・ビジネスロジックの検証**にも転用できる。

## LLMと数学：現在地と限界の正直な整理

### できること

- 既知の定理を組み合わせた**中程度複雑さの証明の構成**
- 正しい問題設定への誘導と**関連する補題の想起**
- 形式証明ライブラリ（Lean/Coq）のコードの補完・修正
- 数値計算の結果に対する**直感的な妥当性チェック**

### できないこと・苦手なこと

- **真に新規な数学的アイデアの発明**（組み合わせの新しさ vs 概念の新しさ）
- 精密な数値計算（LLMは電卓ではない。`3.14159 × 2.71828` を暗算させると誤る）
- 長い証明における**文脈の完全保持**（コンテキスト長の限界）
- 自身の誤りへの**メタ認知**（自信満々に誤った証明を提示する）

```python
# LLMに数値計算をさせる悪い例
prompt = "3.14159265 × 2.71828182845 × 1.41421356 を計算してください"
# → LLMが丸め誤りや桁間違いをする可能性大

# 良い例：計算はコードに任せる
import math
result = math.pi * math.e * math.sqrt(2)
# LLMには「この値の数学的意味」を聞く
```

## 実務エンジニアへの示唆：LLMを「数学的思考のペア」として使う

今回のGPT-5.6の成果が示しているのは、LLMが「数学者の代替」になったということではなく、**「数学的思考の加速装置」として実用フェーズに入った**ということだ。

具体的には：

1. **論文の証明追跡**: arXivの論文で「補題3.2の証明の意味がわからない」→ LLMに「この記法の意味と、証明の直感を日本語で説明して」と投げる
2. **アルゴリズム設計の反証検討**: 新しいアルゴリズムが正しいか疑わしいとき、Devil's Advocateパターンで反例を探させる
3. **既存ライブラリの適用可否判定**: 自分の問題が凸最適化として定式化できるかを、LLMに問題の構造を聞きながら確認する
4. **Lean/Coqコードの補完**: 形式証明のボイラープレートをLLMに生成させ、人間が意味の確認に集中する

これらのユースケースは「LLMが全部やる」のではなく「LLMと人間が役割分担する」設計だ。数学に限らず、**LLMを使ったシステム設計の基本原則もここに収束する**。

## まとめ

| 観点 | 要点 |
|---|---|
| なぜLLMが数学を解けるのか | テスト時計算量増加（内部CoT）・形式証明学習・Verifierとの協調 |
| 凸最適化の特性 | 局所最適=大域最適の保証。ML・OR・制御の基盤理論 |
| プロンプト設計3パターン | `/goal`明示・補題分割・Devil's Advocate反証探索 |
| LLMの限界 | 新規数学概念の発明・精密数値計算・長証明の完全保持 |
| 実務活用の方向性 | 「代替」ではなく「加速装置」。役割分担が鍵 |

LLMが数学という「人間の知性の象徴」ともいえる領域に踏み込んできたことは、エンジニアにとって「ツールとしてどう使いこなすか」を真剣に考えるべきフェーズに入ったことを意味する。今日紹介した3つのプロンプトパターンは、明日から試せる最小の一歩だ。
