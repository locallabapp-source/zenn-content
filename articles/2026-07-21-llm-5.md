## TL;DR

- 複数の LLM を使い分ける「LLMゲートウェイ」は、コスト削減・可用性向上・品質担保の3軸で設計する
- ルーティング戦略は「コストベース」「品質ベース」「レイテンシベース」「フォールバック」「Eval-gated」の5パターンに整理できる
- OSS 実装（Relay / LiteLLM / PortKey）の比較から、自作時のトレードオフが見えてくる

## はじめに

2026年、LLM を本番運用しているチームが直面する共通課題があります。

> **「どのモデルをいつ使えばいい？」**

GPT-4o、Claude Sonnet、Gemini Flash、Qwen3、Kimi K3 ——選択肢は増え続け、コスト・品質・レイテンシのトレードオフは複雑化する一方です。

Ramp が自社の LLMゲートウェイ公開と同時に「内部コストを30%削減した」と発表したように、**ルーティング戦略の設計**はエンジニアリング上の重要テーマになっています。

本記事では、OSS 実装を横断して整理した「モデルルーティングの5パターン」と、それぞれの実装上のポイントを解説します。

## LLMゲートウェイとは

LLMゲートウェイとは、アプリケーションと LLM プロバイダの間に挟まるプロキシ層です。

```
Application
    │
    ▼
┌──────────────────┐
│   LLM Gateway    │  ← ここで routing / fallback / logging
└──────────────────┘
    │         │         │
    ▼         ▼         ▼
 OpenAI    Anthropic  Gemini / OSS
```

主な責務は以下のとおりです。

| 責務 | 説明 |
|---|---|
| ルーティング | リクエスト内容・コスト・品質に応じてモデルを選択 |
| フォールバック | 障害・レートリミット時に代替モデルへ切り替え |
| コスト管理 | トークン使用量の記録・上限設定 |
| 観測性 | レイテンシ・エラー率のログ収集 |
| 認証 | API キーの一元管理（アプリ側に各プロバイダキーを持たせない） |

## ルーティングパターン5選

### パターン1: コストベース・ルーティング

最も単純かつ効果的なパターンです。リクエストの「重要度」や「複雑度」に応じて安価なモデルを優先します。

```python
# 概念コード（疑似コード）
COST_TIER = {
    "flash": ["gemini-flash", "claude-haiku"],     # < $0.001/1K tokens
    "standard": ["gpt-4o-mini", "claude-sonnet"],  # < $0.01/1K tokens
    "premium": ["gpt-4o", "claude-opus"],           # > $0.01/1K tokens
}

def route_by_cost(task_complexity: str) -> str:
    if task_complexity == "simple":   # 要約・分類・短文生成
        return COST_TIER["flash"][0]
    elif task_complexity == "medium": # コード補完・QA
        return COST_TIER["standard"][0]
    else:                             # 長文推論・コード生成
        return COST_TIER["premium"][0]
```

**実装のポイント:**

- `task_complexity` の判定ロジックが肝。入力トークン数・プロンプトテンプレートの種別・呼び出し元 API エンドポイントで分類するのが実用的
- LiteLLM では `router_strategy="cost-based"` 設定で近い挙動を実現できる
- 「安いモデルで試して品質が足りなければ上位へ」というウォーターフォール型にすると過剰コストを防げる

---

### パターン2: 品質ベース・ルーティング（Eval-gated）

出力品質をリアルタイム評価し、閾値を下回った場合に上位モデルへ再試行します。GitHub の `llmrelay/relay` が実装している戦略です。

```
Request
  │
  ▼
Tier-1 model (安価)
  │
  ▼
[Evaluator] ── score < threshold ──→ Tier-2 model
  │                                       │
  ▼ (score >= threshold)                  ▼
 Return                               Return
```

**評価軸の例:**

- **Faithfulness**: 事実と矛盾しないか（RAG 用途に重要）
- **Completeness**: 質問に全て答えているか
- **Format compliance**: JSON スキーマに準拠しているか

```python
# Evaluator の最小実装（疑似コード）
async def eval_gated_call(prompt: str) -> str:
    response = await call_model("gpt-4o-mini", prompt)
    score = await evaluate(response, criteria=["completeness", "format"])
    
    if score >= 0.85:
        return response
    
    # 閾値未満 → 上位モデルへエスカレーション
    return await call_model("gpt-4o", prompt)
```

**実装のポイント:**

- Evaluator 自体も LLM で実装できる（メタ評価）が、コスト増になるため単純な正規表現チェック・JSON バリデーションを先行させる
- 再試行回数の上限（max_retries=2 等）を設けないと無限ループリスクがある
- [RAGAS](https://github.com/explodinggradients/ragas) や [DeepEval](https://github.com/confident-ai/deepeval) など OSS 評価フレームワークをスコアリングに利用できる（MIT License）

---

### パターン3: レイテンシベース・ルーティング

レスポンスタイムを優先したいユースケース（チャット UI・リアルタイム音声対話）向けです。複数モデルへ並列リクエストを投げ、最初に返ってきた結果を採用する「First-wins」戦略と、ヘルスチェックで最速モデルを事前選択する戦略があります。

```
┌───────────────────────────────────┐
│           Parallel Dispatch       │
│                                   │
│  ┌─────────┐  ┌─────────┐         │
│  │ Model A │  │ Model B │  ...    │
│  └────┬────┘  └────┬────┘         │
│       │             │             │
│       └──── race ───┘             │
│               │                   │
│          First-wins               │
└───────────────────────────────────┘
```

**実装のポイント:**

- 採用されなかったリクエストは適切にキャンセルする（HTTP streaming の場合は `abort()` / asyncio の `CancelledError` ハンドリング）
- コスト増になるため、SLA の厳しいエンドポイントにのみ適用する
- ヘルスチェック型の場合、過去 N リクエストの p50/p95 を Prometheus 等で記録し動的に重み付けする

---

### パターン4: フォールバック・チェーン

最も信頼性重視の設計です。プライマリモデルが失敗（エラー・タイムアウト・レートリミット）した際に、定義済みのチェーンを順番に試みます。

```yaml
# LiteLLM router_config.yaml イメージ
model_list:
  - model_name: production-llm
    litellm_params:
      model: gpt-4o
      timeout: 10

  - model_name: fallback-1
    litellm_params:
      model: claude-sonnet-3-7
      timeout: 15

  - model_name: fallback-2
    litellm_params:
      model: gemini/gemini-2-flash
      timeout: 20

fallbacks:
  - {"production-llm": ["fallback-1", "fallback-2"]}
```

LiteLLM（MIT License）はこの設定を数行で実現できます。

**実装のポイント:**

- フォールバック先は「コスト」ではなく「信頼性」で選ぶ。OpenAI が落ちているなら Anthropic、Anthropic が遅ければ Gemini という異プロバイダ分散が有効
- エラーの種別を区別する。`RateLimitError` はすぐリトライ可、`InvalidRequestError` はリトライ無意味（プロンプト問題）
- 最終フォールバック先にはローカル推論（Ollama / llama.cpp）を置くことで完全な可用性を確保できる

---

### パターン5: コンテキスト長・モダリティベース・ルーティング

入力の特性（長さ・種別）で最適なモデルを選ぶパターンです。

| 入力の特性 | 推奨モデル群 |
|---|---|
| < 4K tokens・テキスト | 軽量モデル（Haiku・Flash 系） |
| 4K–32K tokens | 標準モデル（Sonnet・4o-mini 系） |
| > 32K tokens（長文書類） | 大コンテキストモデル（Gemini 1.5 Pro 等） |
| 画像 + テキスト | Vision 対応モデル（GPT-4o / Gemini Flash 等） |
| コード生成 | コード特化（Qwen-Coder / Claude Sonnet 等） |

```python
def route_by_context(messages: list, has_image: bool = False) -> str:
    total_tokens = estimate_tokens(messages)  # tiktoken 等で概算
    
    if has_image:
        return "gpt-4o"  # vision 必須
    
    if total_tokens > 32_000:
        return "gemini/gemini-1.5-pro"
    elif total_tokens > 4_000:
        return "claude-sonnet-3-7"
    else:
        return "gemini/gemini-2-flash"
```

**実装のポイント:**

- トークン数の事前推定には [tiktoken](https://github.com/openai/tiktoken)（MIT License）が便利。Claude / Gemini は近似値で十分
- 画像・音声・PDF など**モダリティ**の判定をミドルウェアで一元化しておくと、ルーターがシンプルに保てる

## OSS ゲートウェイの比較

主要 OSS ゲートウェイを整理します。

| プロジェクト | ライセンス | 主な特徴 | 自ホスト対応 |
|---|---|---|---|
| [LiteLLM](https://github.com/BerriAI/litellm) | MIT | 100+ モデル統合・Proxy サーバー内蔵・Redis キャッシュ | ✅ |
| [PortKey](https://github.com/portkey-ai/gateway) | MIT | フォールバック・ロードバランシング・ログ UI | ✅ |
| [llmrelay/relay](https://github.com/llmrelay/relay) | MIT | Eval-gated routing・軽量・Go 製 | ✅ |
| [OpenRouter](https://openrouter.ai) | SaaS | 無料枠あり・統一 API・150+ モデル | ❌（マネージド） |

LiteLLM は最も成熟しており、プロダクション採用事例も多数あります。小規模スタートアップが自ホストで始めるなら LiteLLM Proxy が最短経路です。

## 設計するときの考え方

5パターンを組み合わせる際の判断フローを示します。

```
まずフォールバック・チェーンを基盤に置く（可用性確保）
         │
         ▼
コストが課題 → コストベース・ルーティングを追加
品質が課題  → Eval-gated ルーティングを追加
速度が課題  → レイテンシベース・ルーティングを追加
入力多様性  → コンテキスト長ルーティングを追加
```

最初から全部実装しようとするとオーバーエンジニアリングになります。**「フォールバック → コストベース → Eval-gated」の順番で段階的に導入する**のが現実的です。

## まとめ

LLMゲートウェイのルーティング設計は、以下の5パターンで整理できます。

1. **コストベース** — タスク複雑度でモデルを格付け
2. **Eval-gated** — 品質スコアで上位モデルへエスカレーション
3. **レイテンシベース** — 並列競争または動的重み付け
4. **フォールバック・チェーン** — プロバイダ障害への耐性
5. **コンテキスト長・モダリティ** — 入力特性で適材適所

「とにかくコスト削減したい」ならコストベース + フォールバック、「品質を落とせないプロダクション」なら Eval-gated、という使い分けが出発点になるはずです。

LLM の選択肢が増え続ける中、**ゲートウェイ層を整備してルーティングを外部化すること**は、長期的なアーキテクチャの柔軟性を保つ上で投資対効果の高い取り組みです。
