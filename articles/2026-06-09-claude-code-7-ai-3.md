## TL;DR

- Claude Code は「指示の書き方」で品質が激変する
- 7 つのパターンを覚えるだけで曖昧な生成・手戻りが大幅に減る
- コンテキスト管理・制約明示・段階分解が核心

## なぜ Claude Code の使い方に差が出るのか

Claude Code（Anthropic が提供するターミナルネイティブの AI コーディングエージェント）は、単純な「コードを書いて」という指示でも一定のアウトプットを返してくれる。しかし、**返ってくるコードの品質・意図との合致率・リファクタリング耐性**は、プロンプトの書き方によって劇的に変わる。

その理由は Claude Code のアーキテクチャにある。Claude Code は会話履歴・読み込んだファイル・ツール呼び出し結果を「コンテキストウィンドウ」として保持しながら動く。つまり**何をどの順序でどの粒度で渡すか**が、最終出力を決定する最大の変数になる。

本記事では、実際に多くの開発現場で再現性が確認されている **7 つのプロンプトパターン** を、具体的な入力例と期待出力の比較つきで解説する。

---

## 前提: Claude Code の基本動作モデル

パターンに入る前に、Claude Code の動作を簡単に確認しておく。

```
入力 (プロンプト + コンテキスト)
       ↓
Claude 3.x (Sonnet / Opus / Haiku)
       ↓
ツール呼び出し (Read / Write / Bash / WebFetch …)
       ↓
出力 (コード / テキスト / ファイル変更)
```

Claude Code は `/project` コマンドや `CLAUDE.md` を通じてプロジェクト固有のルールを事前にインジェクションできる。また `--continue` フラグで前の会話コンテキストを引き継げる。これらを踏まえてパターンを読んでほしい。

## パターン 1: 「目的 + 制約 + 出力形式」の三点セット

### ❌ 曖昧な指示

```
fetchUser 関数を直してください
```

### ✅ 三点セット

```
【目的】fetchUser が 429 Too Many Requests を受け取ったとき
        指数バックオフでリトライするように修正してほしい

【制約】
- 外部ライブラリを追加しない (fetch API のみ使用)
- TypeScript strict モード
- 最大 3 回リトライ、最大待機 8 秒

【出力形式】
- 修正後の関数コードのみ (説明不要)
- 行末コメントで変更箇所を 1 行注記
```

**なぜ効くか**: 目的だけだと Claude は「どんなリトライ戦略でもよい」と判断し、axios や p-retry を import することがある。制約で選択肢を絞ると意図通りの実装が一発で出る確率が上がる。

---

## パターン 2: 既存コードをアンカーにする「このコードを拡張して」方式

長大なコードベースに対して「新機能を追加して」と抽象的に頼むと、Claude Code は既存の命名規則・エラーハンドリングスタイルを無視した実装を生成することがある。

**有効な指示パターン**:

```
下記の createOrder 関数と同じスタイル・エラーハンドリングパターンで、
cancelOrder 関数を新規作成してください。

<createOrder の実装> ... </createOrder>

変更点:
- キャンセル可能条件: status が "pending" または "processing" のみ
- 成功時: { ok: true, cancelledAt: ISO8601 } を返す
- 失敗時: createOrder と同様に AppError を throw
```

ポイントは**参照実装 (createOrder) をコンテキストに含めること**。Claude Code は参照実装のスタイルを「お手本」として扱い、命名・型付け・コメント密度を自動で揃える。

---

## パターン 3: 「ステップ分解」でハルシネーションを抑える

複雑な変更を 1 プロンプトで頼むと、Claude Code は中間状態を省略してゴールに直行しようとし、ファイル間の整合性が崩れることがある。

**推奨: タスクをステップに分割して逐次確認**

```
今から 3 ステップで userId を UUID v4 に移行します。
必ず 1 ステップごとに確認を取ってから次へ進んでください。

Step 1: src/types/user.ts の userId を string → `\`${string}-${string}-${string}-${string}-${string}\`` (UUID ブランド型) に変更
Step 2: userId を生成している箇所をすべてリストアップ (変更はしない)
Step 3: リストアップした箇所を crypto.randomUUID() に置換
```

`確認を取ってから次へ進んでください` という一文が、Claude Code に「中間確認ポイントを設ける」行動規範を与える。これにより Step 2 の洗い出しが不完全でも人間がキャッチできる。

---

## パターン 4: 「テストを先に書いて」TDD 駆動プロンプト

実装よりテストを先に出力させると、仕様の曖昧さが露出する。

```
以下の仕様で parseISODuration 関数の Vitest テストケースを先に書いてください。
(実装はまだ書かないで)

仕様:
- "P1Y2M3DT4H5M6S" → { years:1, months:2, days:3, hours:4, minutes:5, seconds:6 }
- "PT30S" → { years:0, ... seconds:30 }
- 不正な文字列 → Error をスロー

テストが確定したら「テスト OK です、実装を書いてください」と声をかけます
```

テストを確認した時点で「秒だけのケースで time designator T が無い場合は?」のような**仕様の穴**が見つかることが多い。実装後に仕様変更するより圧倒的にコストが低い。

---

## パターン 5: 「やってはいけないこと」ネガティブ制約の明示

Claude Code は親切なため、頼んでいない最適化・リファクタリング・型追加をおまけしてくることがある。

```
auth.ts の getSession 関数のバグを修正してください。

【やってはいけないこと】
- 関数シグネチャの変更
- import の追加・削除
- 関数内のコメントの変更
- auth.ts 以外のファイルへの変更

バグ: req.headers['authorization'] が undefined のとき
     TypeError がスローされる → null を返すべき
```

ネガティブ制約を列挙することで、**変更差分の最小化**が保証される。PR レビュー時のノイズが激減する。

---

## パターン 6: コンテキスト圧縮のための「要約を先に作らせる」

長いセッションが続くと Claude Code のコンテキストウィンドウが逼迫し、古いファイル情報を「忘れる」現象が起きる。

**対策: 定期的に「今の状況を要約して」を挟む**

```
今まで変更したファイルと変更内容を箇条書きで要約してください。
フォーマット:
- ファイルパス: 変更概要 (1行)
この要約を次のプロンプトの冒頭に貼り直すので簡潔に。
```

出力例:

```
- src/auth/session.ts: null チェック追加・getSession バグ修正
- src/types/user.ts: userId を UUID ブランド型に変更
- src/db/user.ts: createUser の userId 生成を randomUUID() に置換
```

この要約を新しいセッション冒頭に渡すと、コンテキストロストを防ぎながら精度を維持できる。

---

## パターン 7: 「レビュアーとして見て」批評モードの活用

コードを書いてもらうだけでなく、**書いたコードの問題点を指摘させる**使い方も強力。

```
以下のコードを、シニアバックエンドエンジニアのレビュアーとして見てください。

観点:
1. セキュリティ (OWASP Top 10 基準)
2. パフォーマンス (N+1 クエリ等)
3. エラーハンドリングの網羅性
4. 可読性 (複雑度・命名)

各観点ごとに「問題点 / 重要度 (高中低) / 改善案」の表形式で出力してください。

<レビュー対象コード>
...
</レビュー対象コード>
```

自分で書いたコードのセルフレビューは盲点が生まれやすい。Claude Code を「違う役割」に据えることで視点が切り替わり、素通りしがちな SQL インジェクションリスクや `Promise.all` 欠落を拾ってくれる。

---

## パターンの組み合わせ: 実践シナリオ例

上記 7 パターンは**組み合わせる**ことで真価を発揮する。

**シナリオ: 新しいAPIエンドポイントの追加**

1. **パターン 4 (TDD)** → まずテストを書いてもらい仕様を確定
2. **パターン 2 (アンカー)** → 既存エンドポイントを参照して実装スタイルを統一
3. **パターン 5 (ネガティブ制約)** → 既存ルーティングファイルへの変更を禁止
4. **パターン 7 (レビュー)** → 実装後にセキュリティ・性能を批評させる
5. **パターン 6 (要約)** → 長いセッション中に定期圧縮

---

## よくある失敗パターンと対処

| 失敗 | 原因 | 対処パターン |
|---|---|---|
| 想定外のファイルも変更された | ネガティブ制約がない | パターン 5 |
| リトライ戦略がライブラリ依存になった | 制約なし | パターン 1 |
| 型エラーが別ファイルに波及 | 一括変更指示 | パターン 3 |
| コメントや命名が既存と乖離 | 参照実装なし | パターン 2 |
| セッション後半で古い決定を「忘れた」 | コンテキスト圧迫 | パターン 6 |

## CLAUDE.md で「常設プロンプト」を作る

プロジェクトルートに `CLAUDE.md` を置くと、Claude Code は起動時に自動でそれをコンテキストに読み込む。7 パターンで繰り返し書くことになる定型制約を事前に宣言しておくと効率が上がる。

```markdown
# プロジェクト固有ルール (Claude Code 向け)

## 常に守ること
- TypeScript strict モード・外部ライブラリ追加は明示的に許可された場合のみ
- テストは Vitest + Testing Library
- スタイル変更は css modules のみ (Tailwind 非使用)

## やってはいけないこと
- src/generated/ 配下のファイルの直接編集
- package.json の scripts セクションの変更

## 参照実装
- API ルートのお手本: src/api/users/route.ts
- エラーハンドリングのお手本: src/lib/errors.ts
```

`CLAUDE.md` はチームで共有するリポジトリにコミットすることで、**全員が同じ品質制約のもとで Claude Code を使う**環境が作れる。

---

## まとめ

Claude Code は「賢いが従順」なツールだ。制約を与えなければ「できる最大限」を尽くそうとし、それが余計な変更や想定外のライブラリ採用につながる。逆に 7 パターンの枠組みを与えると、変更範囲が最小化され、レビューコストが下がり、チームの誰が使っても一定品質のアウトプットが出るようになる。

| # | パターン名 | 主な効果 |
|---|---|---|
| 1 | 目的 + 制約 + 出力形式 | 意図通りの実装を一発で |
| 2 | 参照実装アンカー | スタイル統一 |
| 3 | ステップ分解 | 整合性ミス防止 |
| 4 | テスト先行 TDD | 仕様の穴の早期発見 |
| 5 | ネガティブ制約 | 変更差分の最小化 |
| 6 | 要約圧縮 | コンテキストロスト防止 |
| 7 | 批評モード | セルフレビューの盲点解消 |

まずはパターン 1 と 5 から始めてみてほしい。指示の書き方を変えるだけで、今日から Claude Code の出力品質が一段上がるはずだ。
