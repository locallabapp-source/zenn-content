## TL;DR

- AIエージェントの反復タスクはプロンプト+LLM呼び出しを毎回実行しているため、コストが線形に増大する
- 「スキルコンパイル」はLLMの推論過程を確定的なコード(Python関数等)に変換してキャッシュする手法
- 繰り返し実行されるスキルを事前コンパイルすることで、トークン消費を最大94%削減できる

---

## 背景: LLMエージェントのコスト問題

LLMを使ったエージェントシステムが実用段階に入った2025〜2026年、次の問題が顕在化してきた。

**同じ操作を何千回も繰り返すとコストが爆発する**。

たとえば「CSVの特定列を集計する」「ファイルのリネームルールを適用する」「ログからエラーパターンを抽出する」といったタスクは、エージェントが毎回 LLM に投げて推論させている。1回あたりのトークンが 1,000〜5,000 token だとしても、1日100回実行すれば月間で数百万トークンに達する。

さらに深刻なのが**レイテンシ**だ。LLM推論は高速化が進んだとはいえ、I/O待ちを含めると1呼び出しで数百ms〜数秒かかる。リアルタイム性が求められるパイプラインでは無視できない。

この問題を根本から解決するアプローチとして注目されているのが **スキルコンパイル(Skill Compilation)** である。

---

## スキルコンパイルとは何か

スキルコンパイルの核心は一言で言える。

> **「LLMが考えたプロセスを、再利用可能な決定論的コードに変換する」**

通常のエージェント実行フローはこうだ:

```
入力 → [プロンプト構築] → [LLM推論] → [出力パース] → アクション実行
```

毎回LLMが「どうやって解くか」を考え直している。人間で言えば、足し算を毎回「1を足してまた1を足して……」と考えるようなものだ。

スキルコンパイルはこれを以下に変換する:

```
入力 → [コンパイル済みスキル関数を呼び出す] → アクション実行
```

LLMは**最初の1回だけ**推論する。その推論過程をコード(主にPythonの関数)として書き出し、キャッシュする。以降の同種タスクはそのコードを直接実行する。

## 技術的な実装パターン

### パターン1: One-shot コンパイル

最もシンプルな形。タスクの初回実行時にLLMへ「このタスクを実行するPython関数を書いてください」とプロンプトする。

```python
COMPILE_PROMPT = """
以下のタスクを実行するPython関数を生成してください。
関数は引数として入力データを受け取り、結果を返すものにしてください。
コードブロック(```python ... ```)で囲んで返してください。

タスク: {task_description}
入力例: {sample_input}
期待出力: {sample_output}
"""

def compile_skill(task_description: str, sample_input, sample_output) -> str:
    """LLMにスキル関数を生成させ、コード文字列を返す"""
    prompt = COMPILE_PROMPT.format(
        task_description=task_description,
        sample_input=sample_input,
        sample_output=sample_output,
    )
    response = llm.complete(prompt)
    code = extract_code_block(response)
    return code


def run_compiled_skill(code: str, input_data):
    """コンパイル済みスキルを安全なサンドボックスで実行"""
    local_vars = {}
    exec(code, {"__builtins__": safe_builtins}, local_vars)
    skill_fn = local_vars["skill"]
    return skill_fn(input_data)
```

> ⚠️ `exec` を本番利用する場合はサンドボックス(RestrictedPython / Pyodide / Wasm等)を必ず使用すること。

### パターン2: フィードバックループ付きコンパイル

One-shotでコンパイルしたコードが失敗した場合、LLMにエラーを渡して修正させる。

```python
MAX_RETRIES = 3

def compile_with_retry(task_description, sample_input, sample_output):
    code = compile_skill(task_description, sample_input, sample_output)
    
    for attempt in range(MAX_RETRIES):
        try:
            result = run_compiled_skill(code, sample_input)
            if result == sample_output:
                return code  # コンパイル成功
            # 出力不一致の場合も再コンパイル
            error_msg = f"出力不一致: 期待={sample_output}, 実際={result}"
        except Exception as e:
            error_msg = str(e)
        
        # LLMにエラーフィードバックして修正
        fix_prompt = f"以下のコードがエラーになりました。修正してください。\n\nコード:\n{code}\n\nエラー:\n{error_msg}"
        code = extract_code_block(llm.complete(fix_prompt))
    
    return None  # コンパイル失敗 → フォールバックで通常実行
```

### パターン3: スキルライブラリのキャッシュ管理

コンパイル済みスキルをタスクの種別ごとにキャッシュし、類似タスクでは再コンパイルしない。

```python
import hashlib, json, pathlib

CACHE_DIR = pathlib.Path(".skill_cache")
CACHE_DIR.mkdir(exist_ok=True)

def skill_key(task_description: str) -> str:
    """タスク説明からキャッシュキーを生成"""
    return hashlib.sha256(task_description.encode()).hexdigest()[:16]

def get_or_compile(task_description, sample_input, sample_output):
    key = skill_key(task_description)
    cache_file = CACHE_DIR / f"{key}.py"
    
    if cache_file.exists():
        return cache_file.read_text()  # キャッシュヒット: LLM呼び出しゼロ
    
    # キャッシュミス: LLMでコンパイル
    code = compile_with_retry(task_description, sample_input, sample_output)
    if code:
        cache_file.write_text(code)
    return code
```

この設計により、**2回目以降の同種タスクはLLMを一切呼ばない**。ファイルI/Oだけで完結する。

## 94%削減はどこから来るのか

Vivek Haldar氏の検証記事([原典](https://vivekhaldar.com/articles/compiling-an-ai-agent-skill/))では以下のシナリオで計測している。

| フェーズ | トークン消費 |
|---|---|
| コンパイル前(毎回LLM呼び出し) | 1,000 token × N回 |
| コンパイル後(初回のみLLM) | 1,000 token × 1回 + コード実行コスト≒0 |

**N=20回実行の場合**:
- Before: 20,000 tokens
- After: 1,000 tokens (初回コンパイル) + 19回 × ≒0 token
- 削減率: **95%**

現実には「コンパイル失敗してフォールバック」が数回発生するため、最終的な削減率は **約94%** に落ち着いた、というのが実測値の由来だ。

ただし**重要な前提条件がある**:

1. タスクが**繰り返し実行される**こと (1回きりのタスクには効果なし)
2. タスクが**決定論的に解ける**こと (「これについてどう思いますか?」のような対話タスクには不向き)
3. コンパイル済みコードの**入力スキーマが安定している**こと

## 適用できるユースケース vs 適用できないユースケース

### ✅ 適用に向いているタスク

| ユースケース | 理由 |
|---|---|
| CSVの列変換・集計 | ルールが固定・繰り返し実行 |
| ログのパースとエラー分類 | 入力フォーマット安定 |
| ファイルのリネーム/移動ルール適用 | 決定論的 |
| APIレスポンスの正規化 | スキーマが既知 |
| バリデーションロジック | テストケースで検証可能 |

### ❌ 適用に向いていないタスク

| ユースケース | 理由 |
|---|---|
| 自然言語での回答生成 | 決定論的コードに変換できない |
| コンテキスト依存の判断 | 入力の変動幅が大きすぎる |
| 創作・要約・翻訳 | 品質がLLMの推論に依存 |
| 未知フォーマットのデータ処理 | コンパイル時に入力スキーマ不明 |

## 実装時の注意点

### セキュリティ: コード実行サンドボックス

LLMが生成するコードをそのまま `exec` するのは危険だ。以下の対策を必ず取ること:

```python
# ❌ 危険: 無制限なexec
exec(llm_generated_code)

# ✅ 推奨: RestrictedPython を使ったサンドボックス
from RestrictedPython import compile_restricted, safe_globals

byte_code = compile_restricted(llm_generated_code, "<string>", "exec")
exec(byte_code, safe_globals)
```

あるいは **Pyodide(Wasm)** や **Docker コンテナ**での実行分離がより安全だ。特に外部入力をそのままスキルに渡す場合は、コンテナ分離を強く推奨する。

### コンパイル済みコードのバージョン管理

入力スキーマが変わったのにキャッシュが古いままだと、コンパイル済みスキルが誤動作する。対策:

```python
def skill_key(task_description: str, schema_version: str = "v1") -> str:
    payload = f"{task_description}|{schema_version}"
    return hashlib.sha256(payload.encode()).hexdigest()[:16]
```

スキーマ変更時は `schema_version` を上げてキャッシュを無効化する。

### フォールバック戦略

コンパイル失敗やコンパイル済みスキルの実行エラー時は、通常のLLM呼び出しにフォールバックして可用性を担保する。

```python
def execute_task(task_description, input_data, sample_output=None):
    code = get_or_compile(task_description, input_data, sample_output)
    
    if code:
        try:
            return run_compiled_skill(code, input_data)
        except Exception:
            pass  # コンパイル済みスキル失敗
    
    # フォールバック: 通常LLM呼び出し
    return llm_execute(task_description, input_data)
```

## 関連技術との比較

スキルコンパイルに似たアプローチはいくつか存在する。

| 技術 | 概要 | スキルコンパイルとの違い |
|---|---|---|
| **プロンプトキャッシング** (Anthropic/OpenAI) | 同一プロンプトのプレフィックスをAPIサーバー側でキャッシュ | サーバー側キャッシュ・LLM呼び出し自体は発生する |
| **DSPy** (Stanford NLP) | LLMプログラムの最適化フレームワーク | プロンプト最適化が主目的・コード変換ではない |
| **Semantic Kernel のプランナー** | タスクをスキルの組み合わせとして表現 | スキル実装はLLMに委譲・コンパイルの概念なし |
| **スキルコンパイル** | LLM推論をPythonコードに変換・以降はコード実行 | **LLM呼び出しを排除する**点が最大の違い |

プロンプトキャッシングはあくまでトークン処理コストの削減だが、スキルコンパイルは**LLM呼び出し回数そのものをゼロに近づける**。アプローチの根本が異なる。

## 今後の展望: セルフコンパイリングエージェント

スキルコンパイルを発展させると、**エージェントが自律的にスキルライブラリを成長させる**構造が見えてくる。

```
1. 新しいタスクに遭遇 → LLMで実行
2. 同種タスクが N 回以上発生したことを検知
3. 自動でスキルコンパイルをトリガー
4. コンパイル済みスキルをライブラリに追加
5. 以降の同種タスクはコード実行のみ
```

このループを回し続けると、エージェントは**よく使うタスクを徐々にLLMから切り離し**、高速・低コストな決定論的コードで処理するようになる。富士通のMAAF(Multi-Agent Architecture Framework)が「自己進化」として研究しているアーキテクチャとも思想的に共鳴する部分だ。

---

## まとめ

- スキルコンパイルは「LLMが考えたことをコードに落とし込んでキャッシュする」シンプルな発想
- 繰り返し実行される決定論的タスクにおいては、トークン消費を**最大94%**削減できる
- 実装の要点は「安全なコード実行サンドボックス」「キャッシュのバージョン管理」「フォールバック戦略」の3点
- 対話・創作など非決定論的タスクには向かないが、ETL・バリデーション・ログ処理など**定型オペレーション**への適用効果は大きい
- プロンプトキャッシングとは根本的に異なり、LLM呼び出し回数を削減する唯一の手法として今後の注目度は高い

LLMエージェントの運用コストに課題を感じているなら、まず「このエージェントが1日何回同じ種類の推論をしているか」を計測することから始めてみてほしい。驚くほど多くのタスクがコンパイル可能だと気づくはずだ。
